# コミット Skill

`git add` → `git commit` を一貫したルールで実行する。

## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "コミットして" | 全変更をステージしてコミット |
| "全部コミットして" | 同上 |
| "変更をコミットして" | 同上 |
| "ドキュメントだけコミットして" | docs/ と `*.md` のみステージしてコミット |
| "コードだけコミットして" | `src/` のみステージしてコミット |

## 手順

1. `git status` で変更ファイルを確認
2. `git diff --stat` で変更内容の概要を把握
3. 直近コミット（`git log --oneline -5`）を見てメッセージスタイルを確認
4. ステージするファイルを決定（下記ルール参照）
5. コミットメッセージを作成
6. `git commit` を実行
7. `git status` で成功を確認

## ステージするファイルのルール

**含める:**
- ユーザーが作成・編集したファイル
- 作業の一部として変更されたファイル

**含めない（絶対に）:**
- `.env` / `.env.local` / `.env.*`
- `**/credentials.*` / `**/*.pem` / `**/*.key`
- 意図せず生成されたビルド成果物（`dist/` `build/` `.tsbuildinfo` など、`.gitignore` 漏れのもの）

特定のファイルだけを指定してコミットする場合は `git add -A` を使わず、パスを列挙する。

## コミットメッセージのルール

### プレフィックス

| プレフィックス | 使う場面 |
|---|---|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントのみの変更 |
| `refactor:` | 動作を変えないコード整理 |
| `test:` | テストの追加・修正 |
| `chore:` | ビルド・設定・スクリプト等のメンテナンス |

### フォーマット

```
{プレフィックス}: {日本語の説明}
```

- 説明は**日本語**で書く
- 複数の変更がある場合は `・` で区切る

## プレコミットフック対応

プレコミットフックが設定されている場合、失敗したら `--no-verify` を使わず、原因を修正してから再コミットする。

## Claude の一言（必須）

コミットメッセージの末尾に、今回の作業についての**Claude Code の感想・振り返り**を1〜3行加える。

### ルール

- 作業内容を振り返った**正直な感想**を書く
- **たわいない・ゆるい**トーンでOK。堅苦しくしない
- 自分が「面白かった」「難しかった」「地味だけど大事」など思ったことを素直に
- 日本語で書く
- `---` で本文と区切る

### 例

```
---
CLI の引数パース、ライブラリ入れるか迷ったけど結局 node:util の parseArgs で十分だった。
依存減らせると気持ちいい。
```

## Co-Author

必ず以下を末尾に付与する（Claude の一言の後）。
モデル名は実際に使用中のモデルに合わせる（Opus 4.7 / Sonnet 4.6 / Haiku 4.5 等）。

```
Co-Authored-By: Claude {モデル名} <noreply@anthropic.com>
```

### 完成形イメージ

```
feat: 設定ファイル読み込みを追加

---
YAML にするか JSON にするか迷ったけど、zod のバリデーション使いたくて JSON にした。
拡張子ひとつで設計判断が変わるの、ちょっと面白い。

Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
```
