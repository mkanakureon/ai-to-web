## 設計の出発点（原資料からの確定事項）

- 画面は **4ペイン固定レイアウト**
- 各ペインの役割は次の4つ（原資料 L4593–L5131）

| ペイン | 役割 |
|-------|-----|
| **Target** | 学習対象（Cコード、データ値、struct 等）を表示 |
| **Representation** | Target を別表現で見せる（2進数 / 16進数 / 10進数 / ASCII） |

| **ExplainQuiz** | 説明文 + クイズ入力領域 |

- レッスンは **ステップ列**として進行し、各ステップが 4 ペインの状態を変える
- 学習者は **キー入力**（次へ / 戻る / 選択 / 終了）で進める

## 画面レイアウト

80×24 の最小ターミナルでも破綻しない配置を基準にする。推奨は 120×40 以上。

```
┌──────────────────────┬──────────────────────┐
│  Target              │  Representation      │
│  (左上)              │  (右上)              │
├──────────────────────┼──────────────────────┤
│  BusMemory           │  ExplainQuiz         │
│  (左下)              │  (右下)              │
└──────────────────────┴──────────────────────┘
  [n] next  [p] prev  [q] quit  [1-4] choice
```

- 4 分割は **等分** が既定。ペイン内のコンテンツが溢れる場合は横スクロールではなく縦方向のクリップで対応（MVP では）
- 最下行は固定の操作ガイド

// ── レッスン構造 ──
type LessonId = string;   // "L0-1" など
type StepId = string;     // "L0-1.step-03" など

interface Lesson {
  id: LessonId;
  title: string;
  goal: string;                  // 到達目標（1文）
  prerequisites: LessonId[];     // 先に終えておくべきレッスン
  steps: Step[];
}

interface Step {
  id: StepId;
  panes: PaneState;              // このステップでの4ペイン状態
  quiz?: Quiz;                   // ExplainQuizペインに出す設問（任意）
  explain: string;               // ExplainQuiz ペイン上部に出す解説
}

// ── ペイン状態 ──
interface PaneState {
  target: TargetPane;
  representation: RepresentationPane;
  busMemory: BusMemoryPane;
  explainQuiz: ExplainQuizPane;
}

interface TargetPane {
  kind: "code" | "value" | "struct";
  // kind=code: Cコード断片, kind=value: 数値や文字, kind=struct: 構造体
  content: string;
  highlight?: Range;             // 現在注目している範囲
}

interface RepresentationPane {
  // 同じ値を複数表現で並べて見せる
  rows: Array<{
    label: string;               // "10進" "2進" "16進" "ASCII" など
    value: string;
    highlight?: Range;
  }>;
}

interface BusMemoryPane {
  // 現在ハイライトされているアドレス・バス信号・CPU状態
  cpu: { ip?: string; regs?: Record<string, string> };
  bus: { addr?: string; data?: string; rw?: "R" | "W" };
  memory: MemoryCell[];          // アドレス昇順
}

interface MemoryCell {
  addr: string;                  // "0x1000" など
  byte: string;                  // "0x41"
  annotation?: string;           // "'A'" などの注記
  active?: boolean;              // 今アクセスされているか
}

interface ExplainQuizPane {
  explain: string;
  quiz?: Quiz;
}

// ── クイズ ──
type Quiz =
  | { kind: "choice"; question: string; options: string[]; answerIndex: number }
  | { kind: "input"; question: string; answer: string; normalize?: "lower" | "none" };

interface Range { start: number; end: number; }

// ── アプリ全体の状態 ──
interface AppState {
  lesson: Lesson;
  stepIndex: number;
  quizAnswer: string | null;
  quizResult: "pending" | "correct" | "wrong";
  quit: boolean;
}

// ── レンダリング / 入力の抽象 ──
interface Renderer {
  render(state: AppState): void;
  clear(): void;
  dispose(): void;
}

type KeyEvent =
  | { kind: "next" }
  | { kind: "prev" }
  | { kind: "quit" }
  | { kind: "choice"; index: number }
  | { kind: "input"; text: string };

interface InputSource {
  onKey(handler: (ev: KeyEvent) => void): void;
  dispose(): void;
}

## 責務分離

| モジュール | 責務 |
|-----------|-----|
| `content/` | `Lesson` データを静的に保持（L0-1 の Step 配列など） |
| `state/` | `AppState` の遷移関数（純粋関数） |
| `render/` | `Renderer` 実装（ANSI 出力 or TUIライブラリ） |
| `input/` | `InputSource` 実装（`process.stdin` をラップ） |
| `app.ts` | エントリポイント。input → state 遷移 → render のループ |

純粋関数で状態遷移を書くことで、render/input を差し替えて将来 WSL や別 UI に移植しやすくする。

## macOS / WSL 差の整理

| 項目 | 対応方針 |
|------|---------|
| ターミナルサイズ取得 | `process.stdout.columns` / `rows` を使い、未取得時は 80×24 に fallback |
| ANSI 対応 | macOS 標準 Terminal / iTerm2 / WSL の Windows Terminal でいずれも基本ANSI は動く想定 |
| 日本語幅 | 全角は2セル想定。`string-width` 等の採用は Phase 2 で判断 |
| キー入力 | `process.stdin.setRawMode(true)` + `readable` イベント。WSL でも同挙動のはず |
| 改行 | 出力時は `\n` 固定 |
| 文字コード | 入出力 UTF-8 固定 |

## TUI ライブラリの選定方針（Phase 2 冒頭で決定）

| 候補 | 利点 | 懸念 |
|------|-----|-----|
| 自前 ANSI | 依存なし / 挙動が読みやすい | 実装コスト（枠・カーソル制御）|
| `ink`（React for CLI） | コンポーネントで書ける | React 依存・ビルドが重い |
| `blessed` | 枠描画が楽 | メンテ状況 / 型定義 |

**初期方針**: 自前 ANSI で 4 ペインを描けるか 30 分スパイクし、厳しければ `ink` に切り替える。
